SECTION 5 — KEY PROCESS FLOWS (Step-by-Step)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

FLOW A: NEW LEARNER ADMISSION
──────────────────────────────
  1. User registers via signup.php OR Parent adds child in Parent portal
  2. INSERT into users (role_id=1) + INSERT into learner_profiles
  3. Learner/Parent submits application → INSERT admission_applications
     (applicant_id=user_id, school_id, grade_level, status='Submitted')
  4. Admin/School reviews → view_application.php
  5. Admin approves → approve_application.php
     → UPDATE admission_applications.status = 'Accepted'
     → INSERT learner_school_map (learner_id, school_id)
  6. Admin enrolls learner into class → INSERT class_enrollments
     (learner_id, class_id, enrollment_date, status='Active')
  7. Class enrollment triggers fee assignment → INSERT learner_fees
  8. Timetable auto-populates for enrolled courses

FLOW B: DAILY ATTENDANCE
─────────────────────────
  1. Teacher logs into Teacher portal → attendance.php
  2. Selects class from their assigned classes (classes WHERE teacher_id)
  3. System loads all learners in that class (class_enrollments WHERE class_id)
  4. Teacher marks each learner: Present / Absent / Late / Excused
  5. INSERT INTO attendance (learner_id, class_id, date, status)
     UNIQUE constraint prevents duplicate entries per learner+class+date
  6. Parent/Admin can view via view_attendance.php
  7. Notifications can be triggered for Absent status → INSERT notifications

FLOW C: ASSIGNMENT LIFECYCLE
──────────────────────────────
  1. Teacher creates assignment → INSERT assignments (course_id, teacher_id, due_date)
  2. Learners view assignments filtered to their enrolled courses
  3. Learner submits → INSERT assignment_submissions (assignment_id, learner_id)
  4. Teacher grades submission → UPDATE assignment_submissions SET grade, graded_at, status='Graded'
  5. Grade reflected in learner's grades view

FLOW D: EXAM → RESULTS → REPORT CARD
───────────────────────────────────────
  1. Admin/Teacher creates exam → INSERT exam_schedules (course_id, exam_type, total_marks)
  2. After exam: Admin/Teacher enters scores → INSERT learner_results
     (exam_id, learner_id, course_id, score)
  3. Grades computed → INSERT/UPDATE learner_grades
     (learner_id, course_id, midterm_grade, final_grade, overall_grade, grade_point)
  4. Report card generated → generate_report_card.php
     Queries: learner_profiles + learner_grades + learner_results + courses
  5. PDF export → generate_pdf_report.php (uses TCPDF library in /vendor/tcpdf/)

FLOW E: FEE PAYMENT
──────────────────────
  1. Admin sets up fee structure → INSERT fee_structures (fee_type, amount, academic_year)
  2. Fee assigned to learner → INSERT learner_fees (learner_id, fee_type, total_amount)
  3. Invoice generated → INSERT invoices + INSERT invoice_items (referencing fee_structures)
  4. Payment recorded → INSERT payment_transactions (fee_id, amount_paid, method)
  5. UPDATE learner_fees SET paid_amount, payment_status (Unpaid→Partially Paid→Fully Paid)
  6. UPDATE invoices SET status = 'Paid'

FLOW F: TEACHER–LEARNER MESSAGING
───────────────────────────────────
  1. Sender (any role) composes message
  2. INSERT notifications (sender_id, recipient_id, title, message, type)
  3. Recipient portal shows unread count (SELECT WHERE recipient_id AND is_read=0)
  4. Recipient reads → UPDATE notifications SET is_read=1

SECTION 9 — IDENTIFIED ISSUES & FIXES NEEDED

    FIX: Standardise to one (conn/db.php). Remove db1, db2, db4.

  ISSUE 2 — Duplicate school-learner mapping tables
    Both learner_schools AND learner_school_map have identical structure.
    Data is only in learner_school_map.
    FIX: Remove learner_schools. Use only learner_school_map everywhere.

    /Parent/ contains add_school.php, add_teacher.php, add_user.php,
    admin_routing_instructions.txt — admin functionality should not be
    accessible from Parent portal.
    FIX: Remove from /Parent/ or add strict role_id guard at top of each file.

  ISSUE 4 — Dev/test files in production root
    debug_ajax.php, test_db.php, test_ajax.php, test_student_school.php,
    test_student_school_mapping.php, passwords.txt MUST be removed.
    passwords.txt especially is a critical security risk.

  ISSUE 5 — get_spaces.php duplicated at root AND admin level
    FIX: Keep only in admin/ or make a shared /api/ directory.

  ISSUE 6 — class_timetable vs timetable — two timetable tables
    timetable is class-level. class_timetable is per-learner.
    Both exist but usage may overlap. Ensure pages use the correct table.

  ISSUE 7 — Learner ID 0 in learner_profiles
    learner_profiles has learner_id=0. Auto-increment IDs should start at 1.
    FIX: Clean seed data; enforce NOT NULL AUTO_INCREMENT > 0.

  ISSUE 8 — learner_parents has learner_id=0 / parent_id=0
    Referential integrity issue. orphan records.
    FIX: Clean up data; add stricter application validation before insert.

  ISSUE 9 — invoice system incomplete
    fee_structures, invoices, invoice_items exist but no data is seeded and
    the invoices system may not be fully wired to the payment flow.
    FIX: Connect fee_structure → invoice generation flow in admin/fees.php.

    FIX: Add a check (e.g., lock file) so these only run once, then block access.
